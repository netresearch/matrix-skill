Matrix announcements that don’t suck

A guideline for coding agents posting into Matrix rooms. Structured, scannable, no celebration emoji, no walls of text. Designed HTML cards when the layout matters more than the words.

00 — TL;DR

Five rules

  1. One headline, one purpose. A Matrix message is a tweet, not a blog post.
  2. Send formatted_body with the HTML subset, always. body stays as a readable plaintext fallback.
  3. Lists beat paragraphs. If you write “and also…”, start a <ul>.
  4. Code in <pre><code> blocks. Commands, paths, version strings, IDs — every one of them goes in <code>.
  5. When the layout matters more than the words, render an HTML card to PNG. Tables, dashboards, comparisons, hero announcements.

01

What a Matrix message actually is

An m.room.message event is a JSON object. Two fields decide rendering: body (plaintext fallback — required, used by notifications, search, IRC bridges) and formatted_body (HTML, only when format is set).

{
  "msgtype": "m.text",
  "format":  "org.matrix.custom.html",
  "body":    "Release: github-release-skill v0.2.0\nRepo: https://github.com/netresearch/github-release-skill",
  "formatted_body":
    "<p><strong>Release:</strong> <code>github-release-skill</code> v0.2.0<br/>Repo: <a href=\"https://github.com/netresearch/github-release-skill\">netresearch/github-release-skill</a></p>"
}

m.text vs m.notice. Use m.notice for unattended automation. Clients render it visually muted and bots are forbidden from auto-replying — this prevents bot-on-bot loops. Use m.text when an agent posts on behalf of a human who reviewed the message.

release-bot 14:02 m.notice

📦 Release: github-release-skill v0.2.0 — signed annotated tags, CI-gated.

SM
Sebastian Mendel 14:05 m.text

I checked the new flow against three repos — works as advertised. Migration notes follow in a thread.

02

The HTML subset you can rely on

The Matrix spec defines an explicit allow-list. Element, Cinny, FluffyChat all converge on roughly the same set. Treat this as the floor.

Allowed

<p> <br> <hr>block flow
<strong> <em> <b> <i> <u> <del>inline emphasis
<code> <pre>code — class="language-…"
<blockquote>quote prior content
<ul> <ol> <li>lists
<h3>–<h6>prefer h3 for titles
<table> <thead> <tbody> <tr> <th> <td>basic tables only
<a href>https/matrix.to/mxc
<span data-mx-color>matrix-specific color
<span data-mx-spoiler>spoiler with reason
<sub> <sup>rare but allowed
<mx-reply>client-injected on rich replies

Banned (silently stripped)

<script>
<iframe>
<form>
<style>
style="…"
<img src=https://…>
<video>
class="…" (mostly)
Markdown vs HTML. If your skill composes in Markdown, run it through a real converter (markdown-it, commonmark) to produce HTML. Do not send raw Markdown in formatted_body — most clients won’t parse it. Keep Markdown-ish flavor in body only.

03

The skeleton of every announcement

Title prefix, subject, version. One-sentence why-this-matters. At most three sections. Footer with links and install command. Same shape every time — readers learn the rhythm and skim faster.

[icon] [Type tag]: [Subject] [— optional version]
[one-sentence why-this-matters]

[Section heading 1]
- bullet
- bullet

[Section heading 2]
- bullet

[Footer: links / tracking ticket / install command]

Type tags work like [SECURITY] in email subjects. Pick one; do not stack.

04

When to use which element

ElementUse it forAvoid for
<strong>the one word in a sentence the reader must not missevery other word
<code>commands, paths, identifiers, version strings, env vars, JSON keysEnglish nouns
<pre><code>multi-line commands, JSON examples, diff snippetsone-liners
<ul>2–7 unordered itemsa list of one
<ol>numbered steps where order mattersbullets that happen to have numbers
<blockquote>quoting a user, an error, a previous decisionindenting for visual variety
<table>tabular data ≥2 cols × ≥3 rowstwo-column term–definition pairs
<hr>separating an unrelated postscriptbetween every section (use spacing)
<a>every URL — never paste raw URLs“click here” link text

Code blocks: language hints. Element renders <pre><code class="language-bash"> with syntax highlighting. Use it. Common: language-bash, language-python, language-json, language-diff, language-yaml.

05

Iconography & emoji

One glyph at the front of the title, optionally one inline glyph per bullet. Never trailing decoration. Never celebrations. The glyph carries meaning — if it doesn’t, drop it.

🤖botagent-authored announcement
📦releasenew feature version
🔧toolinginfra, build, dev experience
🛡securityCVE, hardening, lockdown
⚠️heads-upbreaking change, deprecation
📋digestweekly / multi-skill roundup
🔬RFCproposal, request for comment
🚑hotfixurgent patch
🔥postmortemincident summary
✨new capabilityuse sparingly
🚀rocketnever. it’s a release, not a launchpad.
🎉partynever. nobody is celebrating.

06

When to render an HTML card to PNG

formatted_body handles prose, lists and short tables. It is bad at:

  • Comparisons (before/after, plan tiers, client X vs Y)
  • Dashboards (test counts, coverage deltas, dependency health)
  • Timelines (release calendars, incident timelines)
  • Anything wider than ~3 columns
  • Color-coded status pills, badges, hero announcement cards

For these: design an HTML card, render to PNG headlessly, upload, post as m.image. The plaintext body becomes the searchable fallback.

# 1. Render
chromium --headless=new --hide-scrollbars \
  --window-size=1200,630 \
  --screenshot=card.png \
  "file://$(pwd)/card.html"

# 2. Upload to homeserver
MXC=$(curl -s -X POST \
  -H "Authorization: Bearer $MATRIX_TOKEN" \
  -H "Content-Type: image/png" \
  --data-binary @card.png \
  "$HOMESERVER/_matrix/media/v3/upload?filename=card.png" \
  | jq -r .content_uri)

# 3. Send m.image
curl -s -X PUT \
  -H "Authorization: Bearer $MATRIX_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"msgtype\":\"m.image\",\"body\":\"…plaintext fallback…\",\"url\":\"$MXC\",\"info\":{\"mimetype\":\"image/png\",\"w\":1200,\"h\":630}}" \
  "$HOMESERVER/_matrix/client/v3/rooms/$ROOM/send/m.room.message/$(uuidgen)"
Why HTML→PNG, not LLM-generated images. HTML/CSS gives pixel-precise control. Templates render deterministically every week — diffs are obvious, regressions catchable. Coding agents are great at HTML and bad at rasters. Lean into the strength.

Always pair the image with text. An m.image alone is hostile to screen readers, push notifications, search, and copy-paste. Either set body to the full plaintext fallback, or follow the image with a one-line m.text containing the install command and links.

07

Threads, mentions, edits, redactions

Thread the long stuff. Headline as parent; extended notes, screenshots, and Q&A in replies via m.relates_to: { rel_type: "m.thread" }.

Mentions go through matrix.to — not plaintext @name:

cc <a href="https://matrix.to/#/@sebastian:netresearch.de">@sebastian</a>

Include an m.mentions block so notifications fire correctly. Reserve { "room": true } for outages.

Edits within ~5 minutes use m.relates_to: { rel_type: "m.replace" }. Older than that — reply instead; edits to old messages are silent.

Redact only when content is wrong-and-harmful (leaked secret, mistargeted ping). Never to “clean up”.

Example 01

New skill announcement

Type tag at the front. Version inline. Two columns of bullets max. Install command in a code block at the bottom. No celebration, no rocket.

B
release-bot 13:09 m.notice

🤖 New skill: github-release-skill v0.2.0

Releases that look simple until they go wrong — burned tag names, lightweight tags without signatures, bypassed CI, or release notes that are just a list of PR titles.

What it does

  • Detects ecosystem (TYPO3, Node.js, Go, PHP, skill repos) and finds all version files
  • Bumps versions consistently across ext_emconf.php, composer.json, package.json, guides.xml
  • Creates release branches, PRs, signed annotated tags — full flow
  • Rewrites auto-generated notes into a narrative description

What it prevents

  • gh release create — blocked by hook (lightweight, bypasses CI)
  • gh release delete — blocked (doesn’t recover burned tags)
  • Tag deletion from remote — blocked, recommends patch version

Install: /install-plugin https://github.com/netresearch/github-release-skill
Repo: github.com/netresearch/github-release-skill

Example 04

Heads-up: breaking change

Always lead with the warning glyph. Always state the migration. Always state the timeline.

SM
Sebastian Mendel 09:18 m.text

⚠️ Heads-up: matrix-skill v2.0.0 — breaking

Drops Python 3.8 (EOL). Minimum is now 3.10.

What changes: matrix-read-e2ee.py now uses match statements internally; install fails on 3.9.

Migration: bump your venv before installing v2.

pyenv install 3.10.14
pyenv local 3.10.14
pip install --upgrade matrix-skill==2.0.0

Timeline: v1.x continues to receive security patches until 2026-12-31.
Help: NRS-4421

Pipeline announcement (multi-skill)
